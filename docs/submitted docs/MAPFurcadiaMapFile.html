MAP - Furcadia Map File

MAP files contain all the data about the dream’s initial look and settings: arrangement of objects, floors and walls within the dream, patch it is going to use, how big the dream is, etc.

MAP files are hybrid files that contain both ASCII and binary data: A ‘\n’ (0x0A)-separated block of dream settings and properties (the head), and a binary block of elements positioned throughout the map (the body).

File Structure (Head)

MAP V01.30 Furcadia\n    - File header & version

height=37\n              - Property

width=24\n               - Property

revision=2288\n          - Property

patcht=1\n               - Property

[PROPERTY]=[VALUE]\n

BODY\n                   - Body header (binary data comes after 0x0A)

(BINARY DATA)

For a list of available properties, see the table below!

File Structure (Body)

The body of a MAP file consists of consecutive lists of each item type, where each such list fills up every tile in the map (thus consisting of Width*Height elements each). The objects in each list go in a top-down;left-to-right fashion: (0,0) ; (0,1) ; (0,2) ; … ; (2,0) ; (2,1) ; …

Item Types and Order

  1. Floors (16bit unsigned short)
  2. Objects (16bit unsigned short)
  3. Walls (8bit unsigned byte):

The walls are ordered in such a way as to have a row of North-Western walls and then a row of North-Eastern walls for the same tile:

NW: 0,0 ; 0,1 ; … ; 0,24

NE: 0,0 ; 0,1 ; … ; 0,24

NW: 2,0 ; 2,1 ; … ; 2,24

NE: 2,0 ; 2,1 ; … ; 2,24

Another way of seeing it is to split each tile in half, where the left half is an even X coordinate and the right half is an odd X coordinate. Then, we can see it as sequence of walls going the following way: 0,0 ; 0,1 ; … ; 1,0 ; 1,1; …

  1. Regions (16bit unsigned short)
  2. Effects (16bit unsigned short)

Available Properties

PROPERTY

TYPE

DESCRIPTION

height

int

Dream height

width

int

Dream width (real width and not doubled)

revision

int

Revision of this dream file (incremented every time you save)

patcht

int

Patch type: 0 = Default, 1 = Custom, 2 = From Server (unused)

patchs

string

Patch source: Relative path to the target patch or empty string if default

name

string

Name of the dream (appears under the uploader name on the portal)

allowjs

int

Allow Join/Summon by the summoner? (1 = Yes, 0 = Start of dream only)

allowlf

int

Allow Lead/Follow in the dream? (1 = Yes, 0 = No)

allowfurl

int

Allow access via DreamURLs? (1 = Yes, 0 = No)

swearfilter

int

Enable vulgarity filter in dream? (1 = Yes, 0 = No)

nowho

int

Should we list all the player names in dream (`who/F4)? (1 = Yes, 0 = No)

forcesittable

int

Allow sitting only on sittable objects? (1 = Yes,  0 = Can sit anywhere)

allowshouts

int

Allow shouts inside the dream? (1 = Yes, 0 = No)

rating

string

Maturity rating/standard of the dream (see below)

allowlarge

int

Allow/Use Dream Package features? (1 = Yes, 0 = No)

notab

int

Prevent the use of TAB key to show nearby names? (1 = Yes, 0 = No)

nonovelty

int

Prevent use of seasonal avatars inside the dream? (1 = Yes, 0 = No)

parentalcontrols

int

Enforce parental controls in the dream? (1 = Yes, 0 = No)

noload

int

Prevent loading of map in the Dream Editor? (1 = Yes, 0 = No)

encoded

int

Is the dream data encoded/encrypted? (1 = Yes, 0 = No)

NOTE: The “noload” and “encoded” flags are normally present in encrypted dreams. Dreams withot encryption in them don’t have either of them.
